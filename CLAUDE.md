# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Personal portfolio site for a Data Engineering Manager, built with **Astro** and deployed to GitHub Pages. Content is data-driven (JSON) and internationalized in three languages: pt-BR, en-US, es-ES. Motion uses GSAP + Lenis. (This repo was migrated from Jekyll to Astro; some legacy files under `assets/js/` and several root `*.md` docs predate the migration.)

Live at: https://donotavio.github.io/cv-site-otavio/

## Build & Development Commands

```bash
npm install                                 # Install dependencies
npm run dev                                 # Local dev server (astro dev)
npm run build                               # Build static site into dist/
npm run preview                             # Preview the production build
npm run check                               # Type-check (astro check)
```

### Data Pipeline Scripts (Python 3.11)

```bash
pip install requests beautifulsoup4         # Python dependencies
python3 scripts/fetch_github_data.py        # Update github_activity.json (needs GITHUB_USERNAME, GITHUB_TOKEN)
python3 scripts/fetch_linkedin_data_enhanced.py  # Update LinkedIn JSON (needs LINKEDIN_PROFILE_URL, LINKEDIN_SESSION_COOKIE)
python3 scripts/translate_projects.py       # Regenerate translated project strings

```

There are no automated tests. Verify changes manually: language switcher, timeline interactions, responsive layout.

## Architecture

### Astro Layer
- `astro.config.mjs` — Astro config (`base: /cv-site-otavio`, static output → `dist/`)
- `src/layouts/BaseLayout.astro` — Base HTML shell
- `src/pages/index.astro` — Home page, composes the section components in order; sibling pages: `pix-observatory`, `data-stack-radar-br`, `brasil-cockpit`, `eleicoes-2026`
- `src/components/*.astro` — one component per home section (Hero, About, ArchitectureFlow, Impact, AiDelivery, EnrichmentMachine, Timeline, Skills, TechStack, Projects, GitHubStats, Blog, Recommendations, Contact, Navigation, …). Each section header carries a `[ NN ]` mono-label index that is **sequential in DOM order** — inserting a section means renumbering the following ones.
- `src/styles/*.css` — `tokens.css` is the color/type source of truth (**never hardcode hex outside it**); plus `reset`, `global`, `layout`, `fonts`, `transitions`, `print`.
- `src/lib/i18n.ts` — build-time `t(key)` helper (imports `pt-BR.json`); renders default text into the HTML at build (SSG) so content is visible without JS and CLS ~0.
- `src/scripts/motion/` — motion engine on **GSAP + Lenis**. Only 5 named patterns are allowed (`resolve`, `reveal-up`, `hairline-draw`, `hover-glow`, `smooth-scroll`); all gated by `motionOk` (respects `prefers-reduced-motion`). Durations/easings in `constants.ts` — no ad-hoc values.

### Runtime i18n / legacy JS
- `assets/js/i18n.js` — runtime language switch: fetches `/assets/i18n/<lang>.json` and swaps `[data-i18n-key]` / `[data-i18n-aria]` text without reload.
- Other `assets/js/*.js` (`main.js`, `content.js`, `timeline.js`, `animations.js`, …) are **legacy pre-Astro modules**; new work belongs in `src/`.

### Data Layer (`assets/data/`)
JSON files that drive all content. Most are auto-generated by Python scripts via GitHub Actions (daily cron at 06:00 UTC):
- `profile.json` — Consolidated profile data (built by `build_profile_data.py`)
- `github_activity.json` — GitHub stats and repos
- `linkedin_profile.json`, `linkedin_recommendations.json` — LinkedIn data
- `projects_extended.json`, `tech_stack.json`, `blog_articles.json` — Content data

### i18n (`assets/i18n/`)
Translation dictionaries: `pt-BR.json`, `en-US.json`, `es-ES.json` (nested objects). Referenced in markup via `data-i18n-key` / `data-i18n-aria`. Build-time text is rendered by `src/lib/i18n.ts` (`t()`); runtime language switching by `assets/js/i18n.js`. **`public/assets` is a symlink to the repo-root `assets/`** — editing `assets/i18n/` updates both the build-time import and the runtime-fetched copy, so there is no duplicate to maintain. Keep keys in parity across the three files.

## Coding Conventions

- Astro components + scoped CSS; vanilla TypeScript in component `<script>` blocks; GSAP + Lenis for motion (no other JS libraries)
- 2-space indentation in `.astro`, CSS, and JS/TS
- kebab-case for CSS classes and data attributes (e.g., `hero-title`, `data-i18n-key`)
- Commit messages use Conventional Commits format (e.g., `feat:`, `fix:`, `chore:`)
- CSS custom properties live in `:root` in `src/styles/tokens.css` — never hardcode hex outside it

## CI/CD

- `.github/workflows/update-profile.yml` runs daily to fetch fresh GitHub/LinkedIn data, translate projects, build `profile.json`, and auto-commit to `assets/data/` and `assets/i18n/`.
- `.github/workflows/build-and-deploy.yml` builds the Astro site and deploys to GitHub Pages (`gh-pages`); the data-pipeline workflows dispatch it after committing `data:` changes.
- Additional data pipelines run on their own schedules (see the macro-cockpit, eleicoes, and worldcup workflow families).

## Sub-projeto: Brasil Cockpit

Executive dashboard of Brazil's main macro KPIs (traffic-light status vs targets,
delta, trend). **Embedded Astro sub-project** (same pattern as PIX Observatory &
Data Stack Radar BR) — see `docs/BRASIL_COCKPIT.md` for the full spec.

```bash
pip install -r ingestion_macro/requirements.txt
python ingestion_macro/collect_sgs.py        # BACEN SGS
python ingestion_macro/collect_caged.py      # Novo CAGED (start='2020-01-01')
python ingestion_macro/collect_ptax.py       # BACEN PTAX (USD, EUR)
python ingestion_macro/collect_ibge.py       # IBGE SIDRA (PIB, PNAD, PIM, PMC, PMS)
python ingestion_macro/collect_market.py     # yfinance (^BVSP)
python transform_macro/silver_macro.py       # Bronze → Silver (full rebuild)
python transform_macro/gold_cockpit.py       # Silver → Gold + assets/data/cockpit.json + historico.json
```

- `ingestion_macro/catalog.py` — single source of truth (SGS/SIDRA/market series, `KPI_META`, `TARGETS`). Never hardcode series codes or targets elsewhere.
- Gold writes **Parquet to `data/gold/` AND JSON to `assets/data/`** (Astro fetches the JSON at runtime; zero LLM in runtime).
- Status tokens `--ck-verde/amarelo/vermelho/neutro` (+ `-ink` text variants) in `src/styles/tokens.css`.
- CI: `.github/workflows/macro-cockpit-{daily,monthly,caged,quarterly}.yml` → collect → silver → gold → commit `data:` → dispatch `build-and-deploy.yml`.

## Sub-projeto: Observatório Eleições 2026

Painel apartidário sobre a *máquina de medição* da eleição (quem mede o
eleitorado, onde, com que rigor, a que custo) + o perfil real do eleitorado +
intenção de voto presidencial e estadual (medição, não previsão). Página
`src/pages/eleicoes-2026.astro` (13 seções). Cinco JSONs em `assets/data/`,
todos consumidos por fetch em runtime (zero LLM):

```bash
pip install -r ingestion_eleicoes/requirements.txt
python ingestion_eleicoes/collect_pesquisas.py     # TSE Pesquisas 2026 → bronze
python transform_eleicoes/gold_eleicoes.py         # → eleicoes_pesquisas.json (métricas + concentração/custo)
python ingestion_eleicoes/collect_eleitorado.py    # TSE perfil eleitorado ATUAL (~2,3 GB, stream) → bronze
python transform_eleicoes/gold_eleitorado.py       # → eleicoes_eleitorado.json (perfil por sexo/idade/escolaridade/UF)
python ingestion_eleicoes/collect_precandidatos.py # Wikipedia PT (agrega pesquisas do TSE) → bronze
python transform_eleicoes/gold_precandidatos.py    # → eleicoes_precandidatos.json (1º/2º turno + panorama presidencial)
python ingestion_eleicoes/collect_estaduais.py     # Wikipedia PT, 1 artigo/UF (throttle anti-429) → bronze
python transform_eleicoes/gold_estaduais.py        # → eleicoes_estaduais.json (governador 1º/2º turno + senador, por UF)
python ingestion_eleicoes/collect_integridade.py   # roster do painel + TSE DivulgaCand (fail-soft) → bronze
python transform_eleicoes/gold_integridade.py      # → eleicoes_integridade.json (situação jurídica por candidato)
python transform_eleicoes/gold_fundo_ipca.py       # → eleicoes_fundo_ipca.json (deflator IPCA do Fundo: múltiplo real vs. nominal)
```

- `ingestion_eleicoes/catalog.py` — fonte única (URLs TSE + Wikipedia, mapa de colunas, normalização de candidatos, sufixos de UF, regras). Nunca hardcodar fora daqui.
- `assets/data/eleicoes_contexto.json` — **curado à mão** de fontes oficiais (cargos, Fundo Eleitoral, calendário, exterior, renovação). Cada bloco traz `fonte`/`fonte_url`. Atualizar quando o TSE publicar números novos (não é pipeline).
- **Fundo Eleitoral — múltiplo real (deflator IPCA):** `eleicoes_fundo_ipca.json` é gerado por `transform_eleicoes/gold_fundo_ipca.py` (companion do contexto, **não** o sobrescreve). Lê os valores curados do fundo em `eleicoes_contexto.json` (não hardcoda valores) e puxa o IPCA (BACEN SGS série 433, mesma série do Brasil Cockpit) via API REST — só `requests`. Deflaciona o FEFC de 2018 para reais do último mês publicado e emite o múltiplo REAL (~1,9×) ao lado do nominal (~2,9×) + `acima_inflacao_pct`. A seção [09] funde o companion no card e degrada para só-nominal se o JSON faltar. Fail-soft: se o BACEN cair, mantém o JSON existente.
- **Intenção de voto (presidencial + estadual):** fonte é a Wikipedia PT que agrega as pesquisas registradas no TSE (o dataset de registro do TSE não traz percentuais). Presidencial = 1 artigo; estadual = 1 artigo por UF (24 das 27 têm; faltam DF/MS/MT). Parsing de wikitables é frágil (2 formatos de partido, cargo pelo H2) → coletores fail-soft por estado/tabela, normalização no catálogo, throttle + retry anti-429 no coletor estadual, e passos de CI com `continue-on-error` (as seções degradam sozinhas). Governador renova em 1º/2º turno; Senado renova 2 de 3 vagas/UF em 2026.
- **Situação jurídica (§13 "integridade"):** `eleicoes_integridade.json` (envelope: `metodologia`, `disclaimer`, `estagios`, `fontes`, `itens_por_candidato`). Enquadramento neutro (estágio processual, presunção de inocência), cor mono + `--wc-gold` só p/ "Ficha Limpa". O coletor lê o roster (presidenciais + governadores) dos JSONs do painel e consulta o **TSE DivulgaCand** — mas os dados oficiais por candidato só existem **após o registro (15/08/2026)**: até lá `TSE_DIVULGACAND_COD_ELEICAO` fica vazio no catálogo e o pipeline sai com `itens_por_candidato` vazio (nunca inferir/alegar sem fonte oficial). Preencher o código da eleição no catálogo quando o TSE publicar.
- Cor: monocromático `--ink*` para dados + `--wc-gold*` para líder/destaque (sem cor partidária). Sem hex hardcoded.
- **Mapa de calor (§3 estaduais):** `buildGeoMap()` injeta `assets/img/maps/brazil-states.svg` (27 paths com `data-uf`) e pinta cada UF pela **margem do líder na corrida a governador (1º turno)** — intensidade em `--wc-gold-ink` (apartidário). Hover usa o tooltip flutuante global (`[data-tip]`) com líder/partido, intenção, margem s/ 2º, tendência e volume de pesquisas; clique sincroniza os painéis. `buildGeoMap` é reusável (`opts.selectable`, `opts.legendId`). O tile-cartograma `buildUfMap()` segue em uso na §5 (volume por UF).
- **Mapa presidencial por região (§2):** `eleicoes_presidencial_regioes.json` é **curado à mão** (não pipeline) de UMA pesquisa citável (Genial/Quaest jul/2026, registro TSE), verificada no texto. `renderPresRegioes()` reusa `buildGeoMap` (selectable:false) pintando cada UF pela margem do líder na SUA região. **Não há presidencial por-UF**: a intenção presidencial agregada (`precandidatos.json`) é só nacional e não existe fonte coletável/completa por estado (só ~10 estados em imagens; a Quaest agrupa Norte+CO → 4 macrorregiões). Bloco traz `instituto/registro_tse/fonte_url` + `disclaimer` de cobertura parcial; degrada oculto (`#el-reg[hidden]`) se o JSON faltar. Atualizar à mão quando sair rodada nova com recorte regional.
- CI: `.github/workflows/eleicoes-pipeline.yml` (pesquisas + pré-candidatos + estaduais + integridade, diário) + `eleicoes-eleitorado-monthly.yml` (eleitorado, mensal — job pesado).
